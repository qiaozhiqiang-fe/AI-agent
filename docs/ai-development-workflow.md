# AI 编码自动化流程

本项目采用“需求拆解 -> 代码生成 -> 自检 -> PR 描述”的 AI 协作流程。目标是让每次改动都有清晰范围、真实验证和可审阅的交付说明。

## 一次任务的标准流程

### 1. 需求拆解

开始写代码前，AI 应先完成以下分析：

- 明确目标：要解决什么问题或新增什么能力。
- 确认影响范围：涉及 `backend/`、`frontend/`、配置、文档还是多端联调。
- 列出实施步骤：按最小可交付路径拆分。
- 定义验收标准：什么结果表示任务完成。
- 识别风险：不清楚、不可逆、需要外部权限或会影响兼容性的地方。

如果需求足够明确，AI 可以直接继续实现；如果继续实现会有明显风险，先用中文问一个关键问题。

### 2. 代码生成

实现阶段遵守以下规则：

- 保持最小改动，不重构无关代码。
- 遵循现有技术栈：后端 Express + TypeScript，前端 Vue 3 + TypeScript + Vite + Tailwind CSS。
- 公共数据结构、接口响应和组件状态使用明确类型。
- 涉及环境变量时同步更新 `.env.example`。
- 涉及用户可见行为时同步考虑错误态、加载态和边界情况。

### 3. 自检

AI 完成修改后应执行与改动匹配的检查。

通用全量检查：

```sh
npm run check
```

后端专项检查：

```sh
cd backend
npm run lint
npm run build
```

前端专项检查：

```sh
cd frontend
npm run lint
npm run build
```

只改文档时，可以不运行构建，但需要检查文档内容是否与项目实际结构一致。

自检结束后还应查看 Git 差异：

```sh
git status --short
git diff --stat
```

如果检查失败，优先修复后重跑；如果无法修复或无法执行，要说明失败命令、关键英文错误和剩余风险。

### 4. PR 描述

PR 描述应基于实际 diff 和真实验证结果生成，不编造未执行的检查。

推荐结构：

```md
## 背景

## 改动内容

## 验证结果

## 风险与影响

## 备注
```

GitHub PR 默认模板位于 `.github/pull_request_template.md`。

默认不要把改动直接推到 `main`。标准做法是：

```sh
git switch -c codex/<topic>
git push origin codex/<topic>
```

然后基于该分支创建 Pull Request，并使用仓库 PR 模板。

## 配套规则

- 项目总规则：`AGENTS.md`
- 后端局部规则：`backend/AGENTS.md`
- 前端局部规则：`frontend/AGENTS.md`
- 代码评审规范：`docs/code-review.md`
- PR 流程说明：`docs/pull-request-process.md`
- Prompt 模板库：`docs/prompts/`

## 可直接复制的任务提示

```md
请按本项目 AI 编码自动化流程执行：

1. 先做需求拆解，列出目标、影响范围、实施步骤、验收标准和风险。
2. 再开始代码修改，保持最小改动，不改无关文件。
3. 修改后执行与改动匹配的自检命令，并保留真实结果。
4. 最后生成 PR 描述，包含背景、改动内容、验证结果、风险与影响、备注。

如果需求不清楚，先问一个关键问题；如果能合理判断，就直接执行。
```
