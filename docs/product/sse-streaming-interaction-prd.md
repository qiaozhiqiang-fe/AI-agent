# SSE 流式交互项目需求文档

## 1. 背景

当前项目计划建设一个基于 SSE 的 AI 流式交互应用。前端负责承载用户输入、流式输出、会话状态和响应式布局；Node 服务负责提供会话接口、SSE 流式转发能力，并接入 DeepSeek 模型。后续会补充 Figma 设计稿和钉钉产品文档，作为视觉、交互和业务规则的进一步约束。

本文档作为第一版产品需求草稿，用于在正式设计稿和详细产品文档进入前，统一 MVP 范围、接口契约、体验标准和验收口径。

## 2. 目标

- 建立一个可用的 AI 流式问答 MVP，支持用户输入问题并实时看到模型增量输出。
- MVP 必须支持多轮上下文，用户可以围绕同一会话连续追问。
- 前端具备稳定的流式消息布局，能处理生成中、完成、失败、重试、空状态等基础状态。
- 后端提供标准化 SSE 接口，屏蔽 DeepSeek API 细节，对前端输出统一事件格式。
- 为后续接入 Figma 设计稿、钉钉产品文档、更多模型参数和会话能力预留扩展点。

## 3. 非目标

- 第一阶段不建设复杂账号体系、团队空间、权限管理和计费能力。
- 第一阶段不实现多模型市场、插件系统、知识库检索或智能体编排。
- 第一阶段不强依赖线上钉钉 API，钉钉产品文档先以仓库内 Markdown 或文本导入方式使用。
- 第一阶段不承诺完整历史会话云端持久化，但必须能在当前会话内支持多轮上下文。

## 4. 用户与场景

### 4.1 目标用户

- 产品、设计、研发同学：希望快速验证 AI 流式交互体验。
- 内部评审人员：希望基于真实流式输出评估布局、交互和模型响应质量。
- 后续业务使用方：希望在钉钉产品文档和设计稿约束下获得更稳定的 AI 交互能力。

### 4.2 核心场景

1. 用户打开页面，看到清晰的输入区、消息区和基础状态。
2. 用户输入问题并提交，页面立即展示用户消息和 AI 生成中状态。
3. AI 回复通过 SSE 增量渲染，用户不需要等待完整响应结束。
4. 流式输出完成后，消息进入完成状态，用户可以继续下一轮提问。
5. 网络断开、接口异常或模型错误时，页面明确展示失败原因和可重试入口。
6. 用户围绕上一轮回答继续追问时，系统能带上必要上下文生成连贯回复。

## 5. MVP 范围

### 5.1 前端范围

- 单页流式聊天界面。
- 用户输入框，支持提交、提交中禁用、防止空内容提交。
- 消息列表，区分用户消息、AI 消息、系统提示和错误提示。
- AI 消息支持 token 级或片段级增量更新。
- 生成中展示加载状态，并在完成后收敛为普通消息。
- 支持取消本次生成或在断流后重试。
- 支持当前会话内多轮上下文传递和展示。
- 移动端和桌面端基础响应式布局，不出现内容溢出、按钮遮挡或消息区不可读。

### 5.2 后端范围

- 提供健康检查接口，用于确认 Node 服务可用。
- 提供 SSE 聊天接口，接收用户输入并转发 DeepSeek 流式响应。
- 后端统一输出 SSE 事件，不把 DeepSeek 原始事件结构直接暴露给前端。
- 后端必须支持多轮上下文输入，能将当前会话必要历史传递给 DeepSeek。
- 支持基础错误处理，包括参数错误、模型调用失败、上游超时和客户端断开。
- 支持通过环境变量配置 DeepSeek API Key、模型名、API 地址和请求超时时间。

### 5.3 文档与协作范围

- Figma 设计稿进入后，补充页面结构、组件状态和视觉规范。
- 钉钉产品文档进入后，必须补充业务流程、话术要求、合规限制、术语、边界条件和评审结论。
- 所有后续需求变更应同步更新本文档或拆分到 `docs/product/` 下的专项文档。

## 6. 功能需求

### 6.1 页面结构

MVP 主页面由以下元素组成：

- 页面容器：承载整个聊天工作区，负责桌面端和移动端的基础响应式布局。
- 顶部栏：展示产品名称、当前会话状态和新建会话入口。
- 消息区：展示空状态、用户消息、AI 消息、系统状态、错误状态和流式内容。
- 底部输入区：包含多行输入框、发送按钮、取消按钮和输入辅助提示。
- 消息级操作区：按消息状态展示重试、复制、继续追问等操作。
- 滚动辅助区：在用户不处于底部时展示“回到底部”入口。

### 6.2 页面容器

- 页面容器进入页面后默认展示一个空会话。
- 页面容器应占满可用视口高度，消息区和输入区不能互相遮挡。
- 桌面端消息区应作为主要阅读区域，输入区固定在底部可预测位置。
- 移动端输入区应避开系统键盘遮挡，发送、取消和重试入口不能被遮住。
- 当请求处于流式生成中时，页面容器进入 `streaming` 状态；生成完成、失败或取消后退出 `streaming` 状态。

### 6.3 顶部栏

- 顶部栏展示产品名称，MVP 可使用项目默认名称或后续产品命名。
- 顶部栏展示当前会话状态，状态文案包括空会话、正在生成、已完成、已取消、生成失败。
- 顶部栏提供新建会话入口。
- 点击新建会话时，如果当前没有生成中的请求，直接清空本地消息并创建新会话。
- 点击新建会话时，如果当前正在生成，先提示用户会中断当前生成；确认后关闭 SSE 连接、保留当前会话历史，并创建新会话。
- 顶部栏不展示 DeepSeek 模型名称、请求耗时、token usage 等技术信息。

### 6.4 空状态

- 当消息列表为空时，消息区展示空状态。
- 空状态应包含简短引导文案，帮助用户理解可以输入问题开始对话。
- 空状态可以提供少量示例问题；点击示例问题后，将示例内容填入输入框，不自动发送。
- 用户开始输入后，空状态可以保留；提交第一条消息后，空状态消失。
- 空状态不应占用输入区空间，也不应阻挡输入框聚焦。

### 6.5 消息列表

- 消息列表按时间正序展示消息。
- 每条消息必须有唯一 `messageId`、`role`、`status`、`content` 和创建时间。
- 用户消息靠近输入行为展示，AI 消息紧随对应用户消息之后展示。
- 消息列表应支持长文本、换行、列表和代码片段的基础可读展示。
- 消息内容不得横向撑破页面；超长单词、URL 或代码应换行或横向滚动。
- 消息列表中的技术调试信息默认不展示给用户。

### 6.6 用户消息

- 用户点击发送后，用户消息立即追加到消息列表。
- 用户消息内容使用用户提交时的文本快照，不受后续输入框编辑影响。
- 用户消息提交成功后，输入框清空并保持可继续输入状态。
- 用户消息发送失败时，不删除已追加的用户消息，应在对应 AI 消息或系统提示中展示失败状态。
- 用户消息 MVP 不提供编辑功能；如后续支持编辑，应作为独立需求处理。

### 6.7 AI 消息

- 用户消息追加后，应立即创建一条 AI 占位消息。
- AI 占位消息初始状态为 `loading`，展示生成中反馈。
- 收到 `start` 事件后，AI 消息绑定服务端返回的 `messageId` 和 `conversationId`。
- 收到 `delta` 事件后，将 `data.content` 追加到当前 AI 消息内容末尾。
- 连续 `delta` 事件应保持顺序追加，不允许覆盖已展示内容。
- 收到 `done` 事件后，AI 消息状态变为 `done`，取消入口隐藏，可展示复制入口。
- 收到 `error` 事件或连接异常后，AI 消息状态变为 `error`，保留已收到内容，并展示错误提示和重试入口。
- AI 消息不展示模型名称、请求耗时、token usage、原始 SSE 数据或上游错误栈。

### 6.8 输入框

- 输入框为多行文本输入，支持中文、英文、数字、标点、换行和粘贴文本。
- 输入框为空或仅包含空白字符时，发送按钮置为不可用。
- 输入内容超过限制时，应提示用户缩短内容；具体字数限制后续可按模型上下文和业务规则确定。
- 输入框聚焦时，不应导致消息区关键内容被遮挡。
- 用户按 Enter 的行为：MVP 默认发送消息；如需要换行，可使用 Shift + Enter。
- 用户粘贴多行文本时，应保留换行。
- 生成中允许继续编辑输入框草稿，但不能再次发送。
- 当前轮生成完成、失败或取消后，如果输入框有草稿，发送按钮按内容有效性恢复可用状态。

### 6.9 发送按钮

- 发送按钮位于输入区，与输入框形成明确绑定关系。
- 输入为空、仅空白、超长、当前轮生成中或请求正在初始化时，发送按钮不可用。
- 点击发送按钮后，立即执行输入校验。
- 校验通过后，创建用户消息和 AI 占位消息，清空输入框，并发起 `/api/chat/stream` 请求。
- 发送按钮触发请求后进入不可重复提交状态，直到当前轮生成完成、失败或取消。
- 如果请求创建失败，发送按钮恢复可用状态，并在消息区展示失败提示。

### 6.10 取消按钮

- 取消按钮仅在当前轮 AI 消息处于 `loading` 或 `streaming` 状态时展示。
- 点击取消按钮后，前端关闭当前 SSE 连接。
- 取消后已生成内容必须保留，AI 消息状态变为 `canceled`。
- 如果取消时尚未收到任何 `delta` 内容，AI 消息仍保留为空的已取消状态，并展示取消提示。
- 取消后输入框和发送按钮按当前草稿内容恢复状态。
- 取消操作不删除用户消息，不自动重试，也不自动创建新会话。

### 6.11 重试按钮

- 重试按钮仅在 AI 消息处于 `error` 状态时展示。
- 点击重试按钮后，复用对应用户消息内容和当前会话上下文重新发起请求。
- MVP 默认追加一条新的 AI 消息承载重试结果，不覆盖原失败消息。
- 重试过程中，原失败消息保持可见，新 AI 消息进入 `loading` 或 `streaming` 状态。
- 重试成功后，新 AI 消息状态变为 `done`；原失败消息保持 `error` 状态，便于用户理解历史。
- 重试失败时，新 AI 消息变为 `error`，仍可继续重试。

### 6.12 复制按钮

- 复制按钮在 AI 消息状态为 `done`、`canceled` 或带有部分内容的 `error` 时展示。
- 点击复制按钮后，将当前消息可见文本复制到剪贴板。
- 复制成功后给出短暂成功反馈。
- 复制失败时给出失败提示，不影响消息状态。
- 空内容 AI 消息不展示复制按钮。

### 6.13 回到底部按钮

- 用户位于消息列表底部附近时，不展示回到底部按钮。
- 用户手动向上滚动并离开底部时，展示回到底部按钮。
- 点击回到底部按钮后，消息列表滚动到最新消息。
- 流式生成中如果用户停留在底部附近，应自动跟随最新 `delta` 内容。
- 流式生成中如果用户主动向上滚动，不应强制滚回底部。

### 6.14 多轮上下文

- MVP 必须支持当前会话内多轮上下文。
- 每次发送消息时，前端应携带当前会话必要历史。
- 上下文至少包含最近的用户消息和 AI 完成消息。
- 已取消消息可作为上下文候选，但后端应根据状态决定是否传给 DeepSeek。
- 失败消息默认不作为有效 AI 回复传给 DeepSeek，避免污染后续回答。
- 后端应控制传给 DeepSeek 的上下文长度，避免无边界传递完整历史导致性能和成本不可控。
- 当上下文被截断或压缩时，前端无需展示技术细节，但后端应保留可排查日志。
- 钉钉产品文档中的业务流程、话术和合规限制进入后，应作为高优先级上下文或系统约束处理。

### 6.15 状态处理

- 初始状态：展示空会话、顶部栏和输入区，发送按钮按输入内容决定是否可用。
- Loading 状态：用户消息已追加，AI 占位消息已创建，尚未收到首个 `delta`。
- Streaming 状态：正在持续接收 `delta`，AI 消息内容增量增长，取消按钮可用。
- Done 状态：收到 `done` 事件，AI 消息完成，用户可以继续下一轮追问。
- Canceled 状态：用户取消后保留已生成内容，并标记为已取消。
- Error 状态：收到 `error` 事件、网络断开或解析失败，展示错误摘要和重试入口。
- Empty 状态：当没有任何会话消息时，页面不应显示空白区域。

### 6.16 错误提示

- 错误提示应靠近发生错误的消息展示。
- 参数错误应提示用户检查输入内容。
- 网络错误应提示用户检查网络或稍后重试。
- DeepSeek 鉴权、限流、超时等上游错误应转换为用户可理解的通用提示。
- 错误提示可以展示稳定错误码，但不展示后端堆栈、密钥、原始上游响应或敏感信息。
- 错误发生后，输入框不应丢失用户已输入但未提交的草稿。

### 6.17 DeepSeek 接入

- 后端通过服务端环境变量读取 `DEEPSEEK_API_KEY`，前端不得接触真实密钥。
- 后端请求 DeepSeek 时启用流式输出。
- 后端应将 DeepSeek 流式响应转换为项目内部 SSE 事件。
- 模型名、API Base URL、超时时间应可配置，便于后续切换模型或环境。
- 前端用户界面不展示 DeepSeek 模型名称、请求耗时、token usage 等技术信息。

## 7. 接口需求

### 7.1 健康检查

```http
GET /api/health
```

成功响应：

```json
{
  "status": "ok"
}
```

### 7.2 发起流式聊天

```http
POST /api/chat/stream
Accept: text/event-stream
Content-Type: application/json
```

请求体：

```json
{
  "message": "请帮我总结这份产品文档",
  "conversationId": "optional-conversation-id",
  "context": [
    {
      "role": "user",
      "content": "上一轮用户问题"
    },
    {
      "role": "assistant",
      "content": "上一轮 AI 回复"
    }
  ]
}
```

字段说明：

- `message`：必填，用户本轮输入。
- `conversationId`：必填或由后端在 `start` 事件中创建并返回，用于标识当前多轮会话。
- `context`：必填，上下文消息列表，MVP 可先由前端传入最近若干轮，后端负责长度控制和状态过滤。

### 7.3 SSE 事件格式

连接建立：

```text
event: start
data: {"messageId":"assistant-message-id","conversationId":"conversation-id"}
```

增量输出：

```text
event: delta
data: {"content":"增量文本"}
```

完成：

```text
event: done
data: {"finishReason":"stop"}
```

失败：

```text
event: error
data: {"code":"MODEL_REQUEST_FAILED","message":"DeepSeek request failed"}
```

心跳：

```text
event: ping
data: {}
```

## 8. 前端交互与布局要求

- 页面主区域优先服务聊天体验，不做营销型落地页。
- 桌面端消息区应占据主要视口，输入区固定在可预测位置。
- 移动端输入区、发送按钮、取消按钮和消息内容不能互相遮挡。
- 长文本、代码块、列表和错误信息应可读，不撑破页面宽度。
- 生成中内容应自然增长，页面滚动策略应避免频繁跳动。
- 用户手动向上滚动查看历史时，不应强制抢回到底部；用户接近底部时可自动跟随最新输出。
- 消息区应清楚表达不同消息状态，但不展示模型名称、耗时、token usage 等技术字段。
- 当前轮生成中时，页面应突出“正在生成”的操作反馈，同时保留取消入口。
- 多轮追问时，应让用户能自然理解上下文连续性，不要求用户重复输入前置信息。
- 失败、取消、重试等操作应尽量贴近对应消息展示，避免用户误操作到其他轮次。
- 输入区、发送按钮和取消按钮应保持稳定位置，不因消息内容增长产生明显跳动。
- 消息级操作只在对应状态下展示，避免用户看到不可执行操作。
- 所有可点击元素应有明确禁用态、加载态或成功/失败反馈。

## 9. 错误与边界

- 用户输入超长时，应在前端提示或由后端返回明确参数错误。
- DeepSeek API Key 缺失时，后端应启动失败或接口返回明确配置错误。
- DeepSeek 上游超时、限流、鉴权失败时，后端应转换为稳定错误码。
- 客户端主动断开时，后端应停止继续请求或转发上游流。
- SSE 中断后，前端应把当前 AI 消息标记为失败或中断，不应假装完成。
- 多轮上下文超长时，后端应进行截断或压缩，并优先保留最近对话和必要业务约束。
- 钉钉产品文档中的业务流程、话术和合规限制进入后，应作为高优先级上下文或系统约束处理。

## 10. 配置需求

后端 `.env.example` 后续应补充：

```env
PORT=3000
DEEPSEEK_API_KEY=replace-with-your-api-key
DEEPSEEK_API_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
DEEPSEEK_REQUEST_TIMEOUT_MS=60000
```

真实 `.env` 不得提交到仓库。

## 11. 数据与埋点

MVP 阶段可先不接入正式埋点系统，但代码结构应便于后续采集：

- 用户发起请求次数。
- 首 token 响应时间。
- 完整响应耗时。
- 取消次数。
- 失败次数和错误码分布。
- 模型 usage 信息可在后端日志或后续内部观测中记录，但不在前端用户界面展示。

## 12. 验收标准

- 启动后端和前端后，用户可以在前端提交问题并看到 AI 回复增量出现。
- 用户可以在同一会话内连续追问，后续回复能基于前文上下文生成。
- 前端可以正确处理 `start`、`delta`、`done`、`error` 和连接异常。
- 生成中、完成、失败、取消、空会话状态都有可见反馈。
- 输入框、发送按钮、取消按钮、重试按钮、复制按钮和回到底部按钮均符合对应展示条件和禁用规则。
- 取消生成后，已生成的部分内容仍保留在消息区。
- 失败重试时，原失败消息保留，新生成结果以新的 AI 消息展示。
- 前端不展示 DeepSeek 模型名称、耗时、token usage 等技术信息。
- DeepSeek API Key 只存在于后端环境变量，不出现在前端构建产物和请求参数中。
- 移动端和桌面端页面无明显布局溢出或关键操作遮挡。
- 断流或上游失败时，用户能看到失败提示并发起重试。
- 钉钉产品文档进入后，其中明确要求的业务流程、话术和合规限制必须能映射到需求或实现约束。
- 文档、环境变量示例和接口契约与实际实现保持一致。

## 13. 分阶段计划

### 阶段一：MVP 流式闭环

- 实现 `/api/chat/stream`。
- 接入 DeepSeek 流式响应。
- 实现当前会话内多轮上下文。
- 实现前端聊天页、输入区、消息区和基础状态，不展示模型名称、耗时和 token usage。
- 完成基础错误处理和手动验证。

### 阶段二：体验增强

- 增加取消、重试、自动滚动策略和更完整的错误提示。
- 按 Figma 设计稿调整视觉、间距、组件状态和响应式布局。
- 完善代码块、长文本、列表等复杂内容展示。
- 进一步优化页面交互逻辑，包括输入节奏、生成中反馈、上下文连续性表达和消息级操作。

### 阶段三：产品文档对齐

- 导入钉钉产品文档到 `docs/product/`。
- 根据钉钉文档补充业务流程、业务话术、合规限制、术语、输入限制和评审规则。
- 将产品规则映射到前端状态、后端参数校验和验收用例。

### 阶段四：可运营与可观测

- 增加会话持久化。
- 增加基础埋点和日志追踪。
- 增加模型参数配置、用量统计和更细粒度错误码。

## 14. 主要风险

- SSE 在代理、部署平台或浏览器环境中可能存在缓冲和断连问题，需要尽早验证目标部署链路。
- DeepSeek 上游错误格式、限流策略和流式协议细节可能影响后端转换逻辑。
- Figma 设计稿进入后，当前 MVP 布局可能需要调整组件结构。
- 钉钉产品文档进入后，可能新增业务限制、敏感词、审计或合规要求。
- MVP 必须支持多轮上下文，单纯由前端传上下文可能无法长期满足质量和安全要求，后续需要服务端会话存储。
